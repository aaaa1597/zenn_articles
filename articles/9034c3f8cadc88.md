## JOINとUNION
複数のテーブルからの情報を結合します。

## 理論編
### はじめに(Introduction)
[Intro to SQL マイクロコース](https://zenn.dev/rg687076/articles/d7cf749c7ba539)では、**INNER JOIN** を使って2つの異なるテーブルの情報を結合する方法を学びました。ここでは、さらにいくつかの種類の **JOIN** と、複数のテーブルから情報を取得するための **UNION** の使い方を学びます。

このレッスンでは、owners と pets という2つの架空のテーブルを使います。

![](https://storage.googleapis.com/zenn-user-upload/fdfcf5d33635-20260215.png)

owners テーブルの各行は、それぞれ異なるペットの飼い主を表しており、ID 列は一意の識別子です。
owners テーブルの Pet_ID 列には、その飼い主が所有しているペットの ID が入っています（この番号は pets テーブルの ID と一致します）。

例えば、

* pets テーブルを見ると、Dr. Harris Bonkers は ID が 1 のペットです。
* owners テーブルを見ると、Aubrey Little は ID が 1 のペットの飼い主です。

この2つの情報を合わせると、Dr. Harris Bonkers の飼い主は Aubrey Little であることが分かります。

同様に、Veronica Dunn には対応する Pet_ID がないので、ペットを飼っていません。また、Pet_ID 列に 5 が存在しないため、Maisie には飼い主がいません。

### JOINs

これまでに学んだように、**INNER JOIN** を使うと、owners テーブルの Pet_ID 列の値と pets テーブルの ID 列の値が一致する行だけを両方のテーブルから取得できます。

![](https://storage.googleapis.com/zenn-user-upload/caa09d5742c0-20260215.png)

この場合、Veronica Dunn と Maisie は結果に含まれません。

では、すべてのペット（飼い主がいるかどうかに関係なく）を含むテーブルを作りたい場合はどうでしょうか？あるいは、両方のテーブルのすべての行を結合したい場合はどうでしょうか？
そのような場合には、別の種類の **JOIN** を使います。

例えば、owners テーブルのすべての行を含むテーブルを作りたい場合は **LEFT JOIN** を使います。ここで「left（左）」とは、クエリの **JOIN** の前に書かれているテーブルのことを指します。（「right（右）」は **JOIN** の後に書かれているテーブルを指します。）

![](https://storage.googleapis.com/zenn-user-upload/7a62a247d806-20260215.png)

上のクエリで **INNER JOIN** を **LEFT JOIN** に置き換えると、両方のテーブルで一致する行に加えて、左側のテーブルのすべての行（一致があるかどうかに関係なく）が返されます。

もし **RIGHT JOIN** を使うと、一致する行に加えて、右側のテーブルのすべての行（一致があるかどうかに関係なく）が返されます。

最後に、**FULL JOIN** は両方のテーブルのすべての行を返します。
一般に、両方のテーブルで一致しない行については、存在しない値の部分が NULL になります。下の図でそれが確認できます。

![](https://storage.googleapis.com/zenn-user-upload/e571bb9f0535-20260215.png)

### UNIONs

これまで見てきたように、**JOIN** はテーブルを横方向（列方向）に結合します。
一方、列を縦方向に連結したい場合は **UNION** を使います。以下の例では、2つのテーブルの Age 列を結合しています。

![](https://storage.googleapis.com/zenn-user-upload/4435ee9fb292-20260215.png)

**UNION** を使う場合、両方の列のデータ型は同じでなければなりませんが、列名は異なっていても構いません。
（例えば、owners テーブルの Age 列と pets テーブルの Pet_Name 列を UNION することはできません。）

**UNION ALL** を使うと、重複する値も含めます。
owners テーブルと pets テーブルの両方に 9 がある場合、結合結果には 9 が2回表示されます。重複を削除したい場合は、UNION ALL を **UNION DISTINCT** に変更します。

```python
# Query to select all stories posted on January 1, 2012, with number of comments
join_query = """
             WITH c AS
             (
             SELECT parent, COUNT(*) as num_comments
             FROM `bigquery-public-data.hacker_news.comments` 
             GROUP BY parent
             )
             SELECT s.id as story_id, s.by, s.title, c.num_comments
             FROM `bigquery-public-data.hacker_news.stories` AS s
             LEFT JOIN c
             ON s.id = c.parent
             WHERE EXTRACT(DATE FROM s.time_ts) = '2012-01-01'
             ORDER BY c.num_comments DESC
             """

# Run the query, and return a pandas DataFrame
join_result = client.query(join_query).result().to_dataframe()
```

#### 1) 質問はどれくらいの時間で回答されるのか？

質問が回答を受け取るまでに通常どれくらい時間がかかるのかを理解したいと考えています。この知識をもとに、Stack Overflow のユーザーに質問をどの順番で表示するかをより良く設計する予定です。

この目的のもと、2018年1月に投稿された質問に焦点を当てた次のクエリを書きました。このクエリは、以下の2列を持つテーブルを返します。

* **q_id** — 質問のID
* **time_to_answer** — 質問が回答を受け取るまでにかかった時間（秒）

以下のクエリを（変更せずに）実行し、出力を確認してください。

```python
first_query = """
              SELECT q.id AS q_id,
                  MIN(TIMESTAMP_DIFF(a.creation_date, q.creation_date, SECOND)) as time_to_answer
              FROM `bigquery-public-data.stackoverflow.posts_questions` AS q
                  INNER JOIN `bigquery-public-data.stackoverflow.posts_answers` AS a
              ON q.id = a.parent_id
              WHERE q.creation_date >= '2018-01-01' and q.creation_date < '2018-02-01'
              GROUP BY q_id
              ORDER BY time_to_answer
              """

first_result = client.query(first_query).result().to_dataframe()
print("Percentage of answered questions: %s%%" % \
```

#### 2) 最初の質問と回答（Part 1）

ユーザーが Stack Overflow を使い始めたときの「最初の体験」に興味があります。

* ユーザーは最初に質問を投稿することの方が多いのか？
* それとも最初に回答を投稿することの方が多いのか？
* サインアップしてから、最初のアクションまでどれくらい時間がかかるのか？

これを調べるために、次の（未完成の）クエリを作成しました。

このクエリは、次の3列を持つテーブルを返します。

* **owner_user_id** — ユーザーID
* **q_creation_date** — そのユーザーが最初に質問を投稿した日時
* **a_creation_date** — そのユーザーが最初に回答を投稿した日時
