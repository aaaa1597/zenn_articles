## クロスバリデーション
モデルをテストするためのより良い方法。

このチュートリアルでは、クロス検証を使用してモデルのパフォーマンスをより適切に測定する方法を学習します。

### 初めに
機械学習は試行錯誤の連続

機械学習は反復的なプロセスです。 どの予測変数を使うか、どのモデルを使うか、モデルにどのような引数を与えるかなど、多くの選択を迫られます。これまでは、バリデーション（検証）データを使ってモデルの質を測定し、その結果に基づいてデータ駆動型で選択を行ってきました。

しかし、この方法には欠点があります。例えば、5,000行のデータセットがあるとします。通常、その20%（1,000行）を検証データとして残しますが、これではモデルのスコアに「偶然性（運）」が入り込む余地があります。つまり、ある1,000行では良い成績を出すモデルが、別の1,000行では不正確になる可能性があるのです。

極端な例として、検証データが1行しかない場合を想像してください。複数のモデルを比較しても、その1点に対して誰が一番良い予測をするかは、ほとんど運ゲーになってしまいます。

一般的に、検証セットが大きければ大きいほど、モデル品質の測定に含まれるランダム性（ノイズ）が減り、信頼性が高まります。 しかし、大きな検証セットを作るには学習データを削る必要があり、学習データが減ればモデルの精度は落ちてしまいます。

### 交差検証（クロスバリデーション）とは？
交差検証では、データの異なるサブセットに対してモデリングプロセスを繰り返し実行し、モデル品質の指標を複数取得します。

例えば、データを全体の20%ずつ、5つの塊に分けるとします。このとき、データを5つの  **「フォールド（fold：折り目、層）」** に分割したと言います。
![](https://storage.googleapis.com/kaggle-media/learn/images/9k60cVA.png)
その後、各フォールドに対して実験を1回ずつ、計5回行います。
- 実験1： 第1フォールドを検証セットとし、それ以外を学習データとして使用。
- 実験2： 第2フォールドを検証セットとし、それ以外を学習データとして使用。
- これを繰り返し、すべてのフォールドを一度ずつ検証セットとして使用します。

最終的に、データ全体の100%がどこかのタイミングで検証に使われることになり、全データに基づいたモデル品質の測定が可能になります。

### いつ交差検証を使うべきか？
交差検証はより正確な測定ができますが、モデルを複数回（フォールドの数だけ）構築するため、実行に時間がかかります。このトレードオフをどう判断すべきでしょうか？
- 小規模なデータセットの場合： 計算の負荷が大きくないため、交差検証を行うべきです。
- 大規模なデータセットの場合： 単一の検証セットで十分です。実行速度が優先されますし、データが十分あれば、わざわざデータを使い回して検証する必要性は低くなります。

「大きい・小さい」の明確な基準はありませんが、モデルの実行が数分以内であれば、交差検証に切り替える価値があります。

あるいは、クロスバリデーションを実行して、各実験のスコアが近いかどうかを確認することもできます。各実験で同じ結果が得られる場合は、単一の検証セットで十分でしょう。

### 実装例
前回のチュートリアルと同じデータを扱います。入力データをXに、出力データをyに読み込みます。
```python
import pandas as pd

# Read the data
data = pd.read_csv('../input/melbourne-housing-snapshot/melb_data.csv')

# Select subset of predictors
cols_to_use = ['Rooms', 'Distance', 'Landsize', 'BuildingArea', 'YearBuilt']
X = data[cols_to_use]

# Select target
y = data.Price
```
次に、欠損値を埋めるための入力器と予測を行うためのランダム フォレスト モデルを使用するパイプラインを定義します。
パイプラインを使わずにクロスバリデーションを行うことは可能ですが、非常に困難です。パイプラインを使用すると、コードが驚くほど簡単になります。

```python
from sklearn.ensemble import RandomForestRegressor
from sklearn.pipeline import Pipeline
from sklearn.impute import SimpleImputer

my_pipeline = Pipeline(steps=[('preprocessor', SimpleImputer()),
                              ('model', RandomForestRegressor(n_estimators=50,
                                                              random_state=0))
                             ])
```
交差検証スコアは、scikit-learnのcross_val_score()関数を使って取得します。フォールド数はcvパラメータで指定します。
```python
from sklearn.model_selection import cross_val_score

# Multiply by -1 since sklearn calculates *negative* MAE
# sklearnは負のMAEを計算するので、-1を掛けます。
scores = -1 * cross_val_score(my_pipeline, X, y,
                              cv=5,
                              scoring='neg_mean_absolute_error')

print("MAE scores:\n", scores)
```
MAE scores:
 [301628.7893587  303164.4782723  287298.331666   236061.84754543
 260383.45111427]
 
スコアリングパラメータは、報告するモデル品質の尺度を選択します。この例では、負の平均絶対誤差（MAE）を選択しました。scikit-learnのドキュメントには、[オプションのリスト](https://scikit-learn.org/stable/modules/model_evaluation.html)が記載されています。

「負のMAE（平均絶対誤差）」を指定するのは、少し意外に感じるかもしれませんね。実は、Scikit-learnには「すべての指標は、数値が高いほど優れていると定義する」という慣習があります。ここで負の値（マイナス）を使うのは、その慣習に合わせるためです。一般的には「負のMAE」なんてまず耳にしませんが、ライブラリ内の統一感を守るための工夫というわけです。

通常、複数のモデルを比較するには、モデルの品質を単一の尺度で評価する必要があります。そのため、複数の実験の平均をとります。
Average MAE score (across experiments):
277707.3795913405

```python
print("Average MAE score (across experiments):")
print(scores.mean())
```

### 結論
クロスバリデーション（交差検証）を用いることで、モデルの品質をより正確に測定できるようになります。さらに、コードがすっきりするというメリットもあります。トレーニング用とバリデーション用のデータを個別に管理する必要がなくなる点に注目してください。したがって、特に小規模なデータセットにおいては、非常に効果的な改善策と言えます。
