## 理論編
### はじめに(Introduction)
前回のレッスンの例で、Keras はモデルの訓練中、各エポックごとの訓練損失（training loss）と検証損失（validation loss）の履歴を保存する、ということを思い出してください。

このレッスンでは、これらの**学習曲線（learning curves）をどのように解釈するか**、そしてそれをどのようにモデル改善に役立てるかを学びます。特に、**アンダーフィッティング（underfitting）** や**オーバーフィッティング（overfitting）** の兆候を学習曲線から読み取り、それを修正するいくつかの方法を見ていきます。

### 学習曲線の解釈(Interpreting the Learning Curves)

訓練データに含まれる情報は、大きく **シグナル（signal）** と **ノイズ（noise）** の2種類に分けて考えることができます。

* **シグナル**：
  新しいデータにも一般化できる部分。モデルが未知データを予測するのに役立つ本質的な情報。
* **ノイズ**：
  訓練データにだけ当てはまるランダムな揺らぎや偶然のパターン。見かけ上は有用に見えても、実際には予測に役立たない部分。

モデルは、訓練セット上の損失を最小化するように重み（パラメータ）を選びながら学習します。しかし、モデルの性能を正しく評価するには、**検証データ（[validation data](https://www.kaggle.com/code/dansbecker/model-validation)）** のような新しいデータで評価する必要があります。

これまで私たちは、エポックごとの訓練損失をプロットしてきました。ここに**検証損失も加えたもの**が「学習曲線」です。
ディープラーニングモデルを効果的に訓練するには、これらを正しく読み取れる必要があります。
![](https://storage.googleapis.com/zenn-user-upload/db8cd3d2c04b-20260219.png)*検証損失は、目に見えないデータに対する予想されるエラーの推定値を提供します。*

#### 検証損失の意味

* **検証損失**は、未知データに対する期待誤差の推定値です。
* 訓練損失は、モデルが**シグナルを学習したとき**にも**ノイズを学習したとき**にも下がります。
* しかし、検証損失が下がるのは、モデルが**シグナルを学習したときだけ**です。

訓練データで学習したノイズは、新しいデータには一般化しません。

つまり：

* シグナルを学習 → 訓練損失も検証損失も下がる
* ノイズを学習 → 訓練損失は下がるが、検証損失は下がらない（曲線の間にギャップができる）

この**ギャップの大きさ**が、モデルがどれだけノイズを学習してしまったかを示しています。

---

#### 理想と現実

理想的には、**すべてのシグナルを学び、ノイズは一切学ばないこと**です。しかし、現実にはそれはほぼ不可能です。

私たちは常にトレードオフをしています：

* シグナルを多く学ばせようとすると、ノイズも多く学んでしまう可能性がある。
* トレードオフが有利なうちは検証損失は下がり続ける。
* しかし、ある時点を超えると、ノイズ学習のコストがシグナル学習の利益を上回り、検証損失が上昇し始める。
![](https://storage.googleapis.com/zenn-user-upload/037561df302c-20260219.png)*アンダーフィッティングとオーバーフィッティング。*

### Underfitting と Overfitting

このトレードオフから、2つの問題が生じます。

🔹 Underfitting（アンダーフィッティング）

  * モデルが十分なシグナルを学習できていない。
  * そのため、損失が本来より高い。

🔹 Overfitting（オーバーフィッティング）

  * モデルがノイズを学びすぎている。
  * 訓練損失は低いが、検証損失が高くなる。

ディープラーニングで重要なのは、この2つの**最適なバランス**を見つけることです。

### Capacity（モデル容量）

モデルの**容量（capacity）** とは、学習できるパターンの大きさや複雑さのことです。

ニューラルネットワークの場合、容量は主に：

* ニューロンの数
* 層の構造（接続の仕方）

によって決まります。

もしモデルがアンダーフィッティングしているようなら、**容量を増やす**べきです。

容量の増やし方には2種類あります：

🔹 幅を広くする（Wider）

  * 各層のユニット数を増やす
  * 比較的線形的な関係を学習しやすい

🔹 深くする（Deeper）

  * 層を増やす
  * より非線形な関係を学習しやすい

どちらが良いかはデータセットによります。
```python
model = keras.Sequential([
    layers.Dense(16, activation='relu'),
    layers.Dense(1),
])

wider = keras.Sequential([
    layers.Dense(32, activation='relu'),
    layers.Dense(1),
])

deeper = keras.Sequential([
    layers.Dense(16, activation='relu'),
    layers.Dense(16, activation='relu'),
    layers.Dense(1),
])
```

### 早期終了(Early Stopping)

モデルがノイズを学び始めると、検証損失は上昇し始めます。
それを防ぐ方法が **Early Stopping（早期終了）** です。

![](https://storage.googleapis.com/zenn-user-upload/08a8cace1498-20260219.png)*検証損失が最小限に抑えられたモデルを維持します。*

検証損失がそれ以上改善しなくなったら、そこで訓練を止めます。
さらに、最良の検証損失を記録した時点の重みに戻すことで、ノイズを学びすぎるのを防ぎます。

メリット：

* 長く学習しすぎることによるオーバーフィットを防ぐ
* 逆に、十分に学習する前に止めてしまうリスクも減らせる
* エポック数は多めに設定しておき、Early Stopping に任せればよい

### Keras での Early Stopping

Keras では **callback（コールバック）** を使って実装します。
コールバックは、ネットワークのトレーニング中に定期的に実行する関数です。早期停止コールバックは、エポックごとに実行されます。(Keras にはさまざまな[便利なコールバック](https://www.tensorflow.org/api_docs/python/tf/keras/callbacks)があらかじめ定義されていますが、[独自のコールバック](https://www.tensorflow.org/api_docs/python/tf/keras/callbacks/LambdaCallback)を定義することもできます。)

```python
from tensorflow.keras.callbacks import EarlyStopping

early_stopping = EarlyStopping(
    min_delta=0.001,  # 改善とみなす最小変化量
    patience=20,      # 何エポック待つか
    restore_best_weights=True,
)
```

パラメータの意味は：

「直近20エポックの間に検証損失が0.001以上改善しなければ、訓練を止め、最良モデルを保持する」
検証損失の増加が過剰適合によるものなのか、それとも単にランダムなバッチ変動によるものなのかを判断するのは難しい場合があります。パラメータを使用することで、いつ検証を停止するかについてある程度の許容範囲を設定できます。

下記 Exampleでわかるように、このコールバックを損失とオプティマイザーとともに fit メソッドに渡します。

### Example – Early Stopping を使った訓練
前回のチュートリアルの例からモデルの開発を続けましょう。ネットワークの容量を増やすだけでなく、過学習を防ぐために早期終了のコールバックも追加します。
ここでもう一度データの準備を行います。
```python
import pandas as pd
from IPython.display import display

red_wine = pd.read_csv('../input/dl-course-data/red-wine.csv')

# Create training and validation splits
df_train = red_wine.sample(frac=0.7, random_state=0)
df_valid = red_wine.drop(df_train.index)
display(df_train.head(4))

# Scale to [0, 1]
max_ = df_train.max(axis=0)
min_ = df_train.min(axis=0)
df_train = (df_train - min_) / (max_ - min_)
df_valid = (df_valid - min_) / (max_ - min_)

# Split features and target
X_train = df_train.drop('quality', axis=1)
X_valid = df_valid.drop('quality', axis=1)
y_train = df_train['quality']
y_valid = df_valid['quality']
```
![](https://storage.googleapis.com/zenn-user-upload/6895540adc5f-20260219.png)
では、ネットワークの容量を増やしてみましょう。かなり大きなネットワークを構築しますが、検証損失が増加の兆候を見せたら、コールバックを使ってトレーニングを停止します。

```python
from tensorflow import keras
from tensorflow.keras import layers, callbacks

early_stopping = callbacks.EarlyStopping(
    min_delta=0.001, # minimium amount of change to count as an improvement
    patience=20, # how many epochs to wait before stopping
    restore_best_weights=True,
)

model = keras.Sequential([
    layers.Dense(512, activation='relu', input_shape=[11]),
    layers.Dense(512, activation='relu'),
    layers.Dense(512, activation='relu'),
    layers.Dense(1),
])
model.compile(
    optimizer='adam',
    loss='mae',
)
```

コールバックを定義したら、fit の引数として追加します（複数指定できるので、リストに記述してください）。早期終了を使用する場合は、必要以上に大きなエポック数を選択してください。
```python
history = model.fit(
    X_train, y_train,
    validation_data=(X_valid, y_valid),
    batch_size=256,
    epochs=500,
    callbacks=[early_stopping], # put your callbacks in a list
    verbose=0,  # turn off training log
)

history_df = pd.DataFrame(history.history)
history_df.loc[:, ['loss', 'val_loss']].plot();
print("Minimum validation loss: {}".format(history_df['val_loss'].min()))
```
Minimum validation loss: 0.09269220381975174

![](https://storage.googleapis.com/zenn-user-upload/9fad540a8064-20260219.png)

そして予想通り、Keras は 500 エポックが完全に完了する前にトレーニングを停止しました。

### 4) トレーニングと解釈(Train and Interpret)
これは早期終了なしのトレーニングと比べて改善されましたか?
![](https://storage.googleapis.com/zenn-user-upload/408ba27b66b7-20260219.png)
早期停止コールバックは、ネットワークが過学習し始めるとすぐにトレーニングを停止しました。さらに、restore_best_weights を含めることで、検証損失が最小となったモデルを維持できます。

Early Stopping を追加して再学習すると：

* 検証損失が悪化し始めたところで自動的に停止
* ノイズを学びすぎる前に止まる
* 最良の重みに戻る

👉 **改善されています。**

理由：

* オーバーフィットする前に止めている
* 検証損失の最小値が、通常学習より良くなることが多い
* 無駄なエポックを回さないため効率的

#### 💡 ポイント

* 容量を増やす → シグナルを学べるようになる
* しかし → ノイズも学びやすくなる
* Early Stopping → ノイズ学習を抑制

もしよければ、`patience` や `min_delta` を変えてみて、どのような違いが出るか確認してください。
* `patience` を小さく → すぐ止まる（ややアンダーフィット寄り）
* `patience` を大きく → 少し粘る（オーバーフィット寄り）
* `min_delta` を大きく → 改善とみなす基準が厳しくなる

このバランス調整が、実務でとても重要になります。
