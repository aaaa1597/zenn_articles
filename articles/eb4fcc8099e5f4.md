## Abstract
Kaggle「高度なSQLの[効率的なクエリの作成](https://www.kaggle.com/code/alexisbcook/writing-efficient-queries)」の翻訳と実行方法の解説

## 効率的なクエリの作成
クエリをより高速に実行し、より少ないデータを使用するように記述します。

## 理論編
### はじめに(Introduction)
クエリの効率がそれほど重要でない場合もあります。例えば、一度しか実行しないクエリや、小さなデータセットに対するクエリであれば、目的の答えさえ得られれば十分です。

しかし、次のような場合はどうでしょうか？
* Webサイトにデータを供給するために何度も実行されるクエリ
  → ユーザーを待たせないために効率的である必要があります。
* 巨大なデータセットに対するクエリ
  → 書き方が悪いと、実行が遅くなり、コストが余計にかかる可能性があります。

多くのデータベースシステムには **query optimizer(クエリオプティマイザ)** があり、クエリをできるだけ効率的に解釈・実行しようとします。
それでも、いくつかの戦略を意識することで、大幅なコスト削減につながる場合があります。

### 便利な関数

異なるクエリの効率を比較するために、次の2つの関数を使用します。
* `show_amount_of_data_scanned()`
  → クエリが使用（スキャン）したデータ量を表示します。
* `show_time_to_run()`
  → クエリの実行時間を表示します。

```python
from google.cloud import bigquery
from time import time

client = bigquery.Client()

def show_amount_of_data_scanned(query):
    # dry_run lets us see how much data the query uses without running it
    dry_run_config = bigquery.QueryJobConfig(dry_run=True)
    query_job = client.query(query, job_config=dry_run_config)
    print('Data processed: {} GB'.format(round(query_job.total_bytes_processed / 10**9, 3)))
    
def show_time_to_run(query):
    time_config = bigquery.QueryJobConfig(use_query_cache=False)
    start = time()
    query_result = client.query(query, job_config=time_config).result()
    end = time()
    print('Time to run: {} seconds'.format(round(end-start, 3)))
```
   

### 効率化の戦略

#### 1) 必要な列だけを SELECT する

つい `SELECT * FROM ...` と書きたくなります。便利で、どの列が必要かを考えなくて済むからです。しかし、これは非常に非効率になりがちです。

特に不要な **テキストフィールド** がある場合は要注意です。テキストフィールドは他の型よりもサイズが大きい傾向があります。

```python
star_query = "SELECT * FROM `bigquery-public-data.github_repos.contents`"
show_amount_of_data_scanned(star_query)

basic_query = "SELECT size, binary FROM `bigquery-public-data.github_repos.contents`"
show_amount_of_data_scanned(basic_query)
```

データ処理量：2682.118 GB
データ処理量：2.531 GB

この例では、スキャンするデータ量が **1000倍も削減** されています。
元データに巨大なテキストフィールドが含まれていたためです。

#### 2) 読み込むデータを減らす

以下の2つのクエリは、どちらもサンフランシスコ市の片道自転車利用時間（秒）の平均を計算します。

```python
more_data_query = """
                  SELECT MIN(start_station_name) AS start_station_name,
                      MIN(end_station_name) AS end_station_name,
                      AVG(duration_sec) AS avg_duration_sec
                  FROM `bigquery-public-data.san_francisco.bikeshare_trips`
                  WHERE start_station_id != end_station_id 
                  GROUP BY start_station_id, end_station_id
                  LIMIT 10
                  """
show_amount_of_data_scanned(more_data_query)

less_data_query = """
                  SELECT start_station_name,
                      end_station_name,
                      AVG(duration_sec) AS avg_duration_sec                  
                  FROM `bigquery-public-data.san_francisco.bikeshare_trips`
                  WHERE start_station_name != end_station_name
                  GROUP BY start_station_name, end_station_name
                  LIMIT 10
                  """
show_amount_of_data_scanned(less_data_query)
```

データ処理量： 0.076 GB
データ処理量： 0.06 GB

駅IDと駅名は **1対1の対応関係** にあるため、クエリでstart_station_id列とend_station_id列を使用する必要はありません。駅IDを含む列のみを使用することで、スキャンするデータ量を削減できます。

#### 3) N:N JOIN を避ける

これまで扱ってきた多くの JOIN は **1:1 JOIN** でした。
これは、各テーブルの各行が、もう一方のテーブルの最大1行とだけ一致するケースです。
![](https://storage.googleapis.com/zenn-user-upload/5766b7fca845-20260215.png)

次に **N:1 JOIN** があります。
片方のテーブルの1行が、もう一方のテーブルの複数行と一致する場合です。
![](https://storage.googleapis.com/zenn-user-upload/94ba75394f70-20260215.png)

最後に **N:N JOIN** があります。
片方のテーブルの複数行が、もう一方のテーブルの複数行と一致するケースです。一般に、他の条件が同じであれば、N:N JOIN は結合後のテーブルの行数が非常に多くなります。
![](https://storage.googleapis.com/zenn-user-upload/da3e0349ccdd-20260215.png)

### 実例（GitHub データ）

以下の2つのクエリは、複数のGitHubリポジトリについて

* 異なるコミッター数
* ファイル数

を数えています。

#### 大きな N:N JOIN を使ったクエリ
```python
big_join_query = """
                 SELECT repo,
                     COUNT(DISTINCT c.committer.name) as num_committers,
                     COUNT(DISTINCT f.id) AS num_files
                 FROM `bigquery-public-data.github_repos.commits` AS c,
                     UNNEST(c.repo_name) AS repo
                 INNER JOIN `bigquery-public-data.github_repos.files` AS f
                     ON f.repo_name = repo
                 WHERE f.repo_name IN ( 'tensorflow/tensorflow', 'facebook/react', 'twbs/bootstrap', 'apple/swift', 'Microsoft/vscode', 'torvalds/linux')
                 GROUP BY repo
                 ORDER BY repo
                 """
show_time_to_run(big_join_query)

small_join_query = """
                   WITH commits AS
                   (
                   SELECT COUNT(DISTINCT committer.name) AS num_committers, repo
                   FROM `bigquery-public-data.github_repos.commits`,
                       UNNEST(repo_name) as repo
                   WHERE repo IN ( 'tensorflow/tensorflow', 'facebook/react', 'twbs/bootstrap', 'apple/swift', 'Microsoft/vscode', 'torvalds/linux')
                   GROUP BY repo
                   ),
                   files AS 
                   (
                   SELECT COUNT(DISTINCT id) AS num_files, repo_name as repo
                   FROM `bigquery-public-data.github_repos.files`
                   WHERE repo_name IN ( 'tensorflow/tensorflow', 'facebook/react', 'twbs/bootstrap', 'apple/swift', 'Microsoft/vscode', 'torvalds/linux')
                   GROUP BY repo
                   )
                   SELECT commits.repo, commits.num_committers, files.num_files
                   FROM commits 
                   INNER JOIN files
                       ON commits.repo = files.repo
                   ORDER BY repo
                   """

show_time_to_run(small_join_query)
```
実行時間: 13.028 seconds
実行時間: 4.413 seconds

最初のクエリには大きなN:N結合が含まれています。クエリを書き直して結合のサイズを小さくすると、実行速度が大幅に向上します。

これらの戦略や、さらに多くのテクニックについては、Google BigQuery の包括的なガイドで詳しく説明されています。より効率的なクエリの書き方を学びたい場合や、BigQuery についてさらに理解を深めたい場合は、ぜひ参照してみてください。

### 演習(Exercises)

#### 1) あなたは Pet Costumes International で働いています。
今日の午後、3つのクエリを書く必要があります。

3つすべてについて、**動くバージョンを書く時間はあります**が、**最適化についてじっくり考えられるのは1つだけ**です。

どのクエリを最適化するのが最も価値があるでしょうか？

1. あるソフトウェアエンジニアが、出荷部門向けのアプリを作りました。
どの商品を出荷する必要があるのか、そして倉庫のどの通路（aisle）に行けばよいのかを確認するためのものです。
彼女はあなたにクエリを書くよう依頼しています。
このクエリでは、`orders` テーブル、`shipments` テーブル、`warehouseLocation` テーブルのデータを使います。
出荷部門の従業員はタブレットでこのアプリを開き、**リフレッシュボタンを押すたびに**、あなたのクエリ結果がきれいなインターフェース上に表示され、どのコスチュームをどこに送ればよいか確認します。

2. CEO が、すべての顧客レビューと苦情の一覧を欲しがっています。
それらは都合よく `reviews` テーブル1つに保存されています。
レビューの中にはとても長いものもあります……
というのも、お客様はオウム用の海賊コスチュームが大好きで、どれだけ可愛いか延々と書き続けてしまうからです。

3. 最近、犬の飼い主はこれまで以上に愛犬を守ろうとしています。
そこでエンジニアリング部門は、GPSトラッカーと無線通信デバイスを埋め込んだコスチュームを開発しました。
それらは **1秒ごとに** コスチュームの座標をデータベースへ送信します。
そして、飼い主が自分の犬（正確には犬が着ているコスチューム）の現在位置を確認できるウェブサイトがあります。
このサービスを機能させるには、**ある特定の人間が所有しているすべてのコスチュームについて、最新の位置を表示するクエリ**が必要です。
このクエリでは `CostumeLocations` テーブルと `CostumeOwners` テーブルのデータを使用します。

最も効率的に書くべきなのはどれでしょうか？

下の `query_to_optimize` 変数に、1・2・3 のいずれかを整数で設定してください。

```python
# Fill in your answer
query_to_optimize = 3

# Check your answer
q_1.check()
```

##### なぜ 3 なのか

コスチュームごとに 1秒ごと にデータが送信されるため、このクエリは（圧倒的に）最も大量のデータを扱う可能性があります。

さらに、このクエリは継続的に実行されます。

そのため、このクエリを効率よく書くことができれば、繰り返し実行されるたびに効果（時間やコストの削減）が積み重なっていきます。

##### なぜ 1 ではないのか

これは最適化する価値が 2番目に高い クエリです。

繰り返し実行されますし、テーブルの結合（JOIN）を含んでいます。
JOIN は、クエリをより効率的にできるポイントになりやすい部分です。

しかし、データ量の規模という点では 3 ほどではありません。

##### なぜ 2 ではないのか

これは 一度だけ実行される 可能性が高いクエリです。

そのため、実行に数秒余計にかかったり、数セント多くコストがかかったとしても、大きな問題にはなりません。

また、JOIN も含まれていません。
レビューというテキストフィールドはありますが、それは取得したいデータそのものです。

したがって、計算量を減らすために SELECT から除外することもできません。


### 2) ミッツィーをもっと見つけやすく！(Make it easier to find Mitzie!)

次の2つのテーブルがあります。
![](https://storage.googleapis.com/zenn-user-upload/10b0891dfdad-20260216.png)

* **CostumeLocations テーブル**
  データベース内のすべてのペット用コスチュームについて、タイムスタンプ付きのGPSデータを示します。
  `CostumeID` は各コスチュームの一意な識別子です。

* **CostumeOwners テーブル**
  各コスチュームの所有者を示します。
  `OwnerID` 列には、それぞれの（人間の）所有者を識別する一意のIDが入っています。

注意点：

* 1人の所有者が複数のコスチュームを持つことができます。
* 1つのコスチュームに複数の所有者がいることもあります（同じ世帯の複数人がアクセスできるようにするため）。

さて、特定のペット1匹の現在位置を取得したいとします。

犬の **Mitzie（ミッツィー）** はリスを追いかけて逃げてしまいましたが、幸いにもホットドッグのコスチュームを着ているところを最後に目撃されています！

Mitzie の所有者の1人（OwnerID は `MitzieOwnerID`）がウェブサイトにログインし、自分が所有するすべてのコスチュームの最新位置を取得します。

現在は、次のクエリを実行して情報を取得しています：

```sql
WITH LocationsAndOwners AS 
(
SELECT * 
FROM CostumeOwners co INNER JOIN CostumeLocations cl
   ON co.CostumeID = cl.CostumeID
),
LastSeen AS
(
SELECT CostumeID, MAX(Timestamp)
FROM LocationsAndOwners
GROUP BY CostumeID
)
SELECT lo.CostumeID, Location 
FROM LocationsAndOwners lo INNER JOIN LastSeen ls 
    ON lo.Timestamp = ls.Timestamp AND lo.CostumeID = ls.CostumeID
WHERE OwnerID = MitzieOwnerID
```

これを **より速く、あるいはより安く実行する方法**はあるでしょうか？

答え
```sql
WITH CurrentOwnersCostumes AS
(
SELECT CostumeID 
FROM CostumeOwners 
WHERE OwnerID = MitzieOwnerID
),
OwnersCostumesLocations AS
(
SELECT cc.CostumeID, Timestamp, Location 
FROM CurrentOwnersCostumes cc INNER JOIN CostumeLocations cl
    ON cc.CostumeID = cl.CostumeID
),
LastSeen AS
(
SELECT CostumeID, MAX(Timestamp)
FROM OwnersCostumesLocations
GROUP BY CostumeID
)
SELECT ocl.CostumeID, Location 
FROM OwnersCostumesLocations ocl INNER JOIN LastSeen ls 
    ON ocl.timestamp = ls.timestamp AND ocl.CostumeID = ls.costumeID
```
### なぜこれのほうが良いのでしょうか？

大規模な結合（JOIN）を行い、すべてのコスチュームに対して最終タイムスタンプを計算する（`MAX(Timestamp)` を求める）代わりに、**最初の段階で他の所有者の行を除外する**からです。

その結果、その後の各ステップ（たとえば最終タイムスタンプの計算）は、元のクエリで必要だった行数と比べて、**99.999%ほど少ない行数**で処理されることになります。

場合によっては、クエリプランナーがこのような最適化（先に絞り込みを行うこと）に気づいてくれるかもしれません。

しかし、元のクエリの書き方のままだと、大規模データセットでは非常に非効率になる可能性があります。
