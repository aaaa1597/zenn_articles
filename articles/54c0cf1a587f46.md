## Abstract
Kaggle「地理空間データ分析の[データ漏洩](https://www.kaggle.com/code/alexisbcook/data-leakage)」の翻訳と実行方法の解説

## データ漏洩(データリーケージ)
モデルを微妙に台無しにするこの問題を見つけて修正します。

### はじめに
データリークは、**「訓練データにはターゲット（正解）に関する情報が含まれているが、予測を行う際（本番環境）には同様のデータが利用できない」** 場合に発生します。これにより、訓練セットやバリデーションデータでは非常に高いパフォーマンスを示す一方、実際の運用（プロダクション）環境では散々な結果に終わります。

言い換えれば、リークが発生したモデルは「実際にそのモデルを使って意思決定を始めるまでは優秀に見えるが、使い始めた途端に役立たずになる」のです。

### ターゲットリーク と 訓練・テスト間の汚染（Train-Test Contamination）
リークには主に2つのタイプがあります： **ターゲットリーク** と **訓練・テスト間の汚染（Train-Test Contamination）** です。

#### ターゲットリーク (Target Leakage)
ターゲットリークは、「予測を行う時点では手に入らないデータ」が特徴量に含まれているときに起こります。単に「予測に役立つかどうか」ではなく、データが利用可能になる **「タイミングや時系列」** で考えることが重要です。

例：肺炎の予測肺炎になる人を予測したいとします。生データが以下のようになっている場合を考えてみましょう。
|got_pneumonia|age|weight|male|took_antibiotic_medicine|...|
|---|---|---|---|---|---|
|False|65|100|False|False|...|
|False|72|130|True|False|...|
|True|58|100|False|True|...|

肺炎にかかった人は回復するために抗生物質の薬を服用します。生データではこれらの列の間に強い相関関係が見られますが、took_antibiotic_medicine は got_pneumonia の値が決定された後に頻繁に変更されています。これはターゲットの漏洩です。

モデルは、took_antibiotic_medicine の値が False の人は肺炎にかかっていないと判断します。検証データはトレーニング データと同じソースから取得されるため、検証ではパターンが繰り返され、モデルの検証 (またはクロス検証) スコアは高くなります。

しかし、このモデルは、その後現実世界に展開されると非常に不正確になります。なぜなら、肺炎になる患者でさえ、将来の健康状態を予測する必要があるときには、まだ抗生物質を投与されていないからです。

このようなデータ漏洩を防ぐには、目標値が実現された後に更新 (または作成) された変数を除外する必要があります。
![](https://storage.googleapis.com/kaggle-media/learn/images/y7hfTYe.png)


#### 訓練・テスト間の汚染 (Train-Test Contamination)
トレーニング データと検証データを慎重に区別しないと、別のタイプのリークが発生します。
検証は、これまで考慮したことのないデータに対してモデルがどのように機能するかを測定するためのものであることを思い出してください。検証データが前処理の動作に影響を与える場合、このプロセスが微妙に破損する可能性があります。これは、**トレインテスト汚染**と呼ばれることもあります。

たとえば、train_test_split() を呼び出す前に前処理 (欠損値に対する入力のフィッティングなど) を実行するとします。最終結果はどうなるでしょうか? モデルの検証スコアは良好で、モデルに対する信頼度は高いものの、実際に展開して意思決定を行うとパフォーマンスが低下する可能性があります。
**その結果**、バリデーションデータの情報が訓練プロセスに紛れ込むため、テストスコアは良くなりますが、未知のデータに対する汎化性能は低くなります。
より複雑な機能エンジニアリングを実行すると、この問題はさらに微妙 (そして危険) になります。
**対策**: scikit-learnの Pipeline（パイプライン） を活用しましょう。特に交差検証（Cross-validation）を行う際は、パイプラインの中で前処理を行うことが不可欠です。

### 実践例：
この例では、ターゲットの漏洩を検出して除去する 1 つの方法を学習します。
クレジットカード申請に関するデータセットを使用し、基本的なデータ設定コードは省略します。最終結果として、各クレジットカード申請に関する情報が DataFrame X に保存されます。これを使用して、シリーズ y でどの申請が承認されたかを予測します。
```python
import pandas as pd

# Read the data
data = pd.read_csv('../input/aer-credit-card-data/AER_credit_card_data.csv', 
                   true_values = ['yes'], false_values = ['no'])

# Select target
y = data.card

# Select predictors
X = data.drop(['card'], axis=1)

print("Number of rows in the dataset:", X.shape[0])
X.head()
```
Number of rows in the dataset: 1319

これは小さなデータセットなので、モデルの品質を正確に測定するためにクロス検証を使用します。
```python
from sklearn.pipeline import make_pipeline
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import cross_val_score

# Since there is no preprocessing, we don't need a pipeline (used anyway as best practice!)
my_pipeline = make_pipeline(RandomForestClassifier(n_estimators=100))
cv_scores = cross_val_score(my_pipeline, X, y, 
                            cv=5,
                            scoring='accuracy')

print("Cross-validation accuracy: %f" % cv_scores.mean())
```
Cross-validation accuracy: 0.981052

クレジットカードの承認予測1,319件のデータを使って、カードの申請が承認されるかを予測します。まず、何も考えずに全データを使って精度を確認すると：Cross-validation accuracy: 0.981052 **精度98%** という驚異的な数字が出ました。経験上、これほど高い精度が出るのは稀であり、ターゲットリークを疑うべきです。

- カード: クレジットカードの申請が承認された場合は 1、承認されなかった場合は 0。
- 報告数: 重大な中傷報告の数
- 年齢: n歳プラス12分の1歳
- 収入：年収（10,000で割ったもの）
- シェア：クレジットカードの月々の支出額と年間収入の比率
- 支出: クレジットカードの月平均支出
- 所有者: 自宅を所有している場合は1、賃貸の場合は0。
- selfempl: 自営業の場合は1、そうでない場合は0。
- 扶養家族：1 + 扶養家族の数
- 月数: 現在の住所に住んでいる月数
- majorcards: 保有している主要クレジットカードの数
- アクティブ: アクティブなクレジットアカウントの数

データを詳しく見ると、expenditure（支出額）という項目が怪しいことがわかります。これは「このカードでの支出」でしょうか？それとも「申請前の他カードでの支出」でしょうか？

この時点で、基本的なデータの比較が非常に役立ちます。


```python
expenditures_cardholders = X.expenditure[y]
expenditures_noncardholders = X.expenditure[~y]

print('Fraction of those who did not receive a card and had no expenditures: %.2f' \
      %((expenditures_noncardholders == 0).mean()))
print('Fraction of those who received a card and had no expenditures: %.2f' \
      %(( expenditures_cardholders == 0).mean()))
```
Fraction of those who did not receive a card and had no expenditures(カードを受け取らず、支出がなかった人の割合)：1.00
Fraction of those who received a card and had no expenditures(カードを受け取ったが支出がなかった人の割合)：0.02

上に示されているように、カードを受け取らなかった人は全員支出がなかったのに対し、カードを受け取った人のうち支出がなかったのはわずか 2% でした。このモデルが高精度であるように見えますが驚くことではありません。なぜなら、これもターゲット漏洩のケースであるように思われ、支出はおそらく申請したカードでの支出を意味していると思われます。

share も除外する必要があります。expenditure(支出)によって部分的に決定されるためです。変数 active と majorcards は少し分かりにくいですが、説明からすると懸念されるものです。難しいですね。ほとんどの場合、データを作成した人を追跡して詳細を調査できない場合は、後悔するより安全策を講じた方が良いでしょう。

ターゲット漏洩のないモデルを次のように実行します。
```python
# Drop leaky predictors from dataset
potential_leaks = ['expenditure', 'share', 'active', 'majorcards']
X2 = X.drop(potential_leaks, axis=1)

# Evaluate the model with leaky predictors removed
cv_scores = cross_val_score(my_pipeline, X2, y, 
                            cv=5,
                            scoring='accuracy')

print("Cross-val accuracy: %f" % cv_scores.mean())
```
Cross-val accuracy: 0.830919

これらを除外して再度モデルを組むと、精度は 83% まで下がりました。スコアが下がったのは残念に感じるかもしれませんが、これこそが「本番環境で実際に発揮される、信頼できる精度」なのです。

### 結論
データリークは、ビジネスにおいて数百万ドル規模の損失を招く可能性があります。データの分離を徹底し、Pipeline を活用することで、訓練・テスト間の汚染を防ぐことができます。
ターゲットリークを防ぐには、注意深い観察と、データ探索（EDA）を組み合わせて不自然な高精度を疑うことが重要です。

### ステップ4：感染の予防

あなたは予測モデルを構築することにしました。モデリング用のデータセットの各行は、手術を受けた個々の患者であり、予測対象（ターゲット）はその患者が「感染症を発症したかどうか」です。
